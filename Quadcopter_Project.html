Design an agent to fly a quadcopter, and then train it using a reinforcement learning algorithm of your choice!

Controlling the Quadcopter¶

We provide a sample agent in the code cell below to show you how to use the sim to control the quadcopter. This agent is even simpler than the sample agent that you'll examine (in agents/policy_search.py) later in this notebook!

The agent controls the quadcopter by setting the revolutions per second on each of its four rotors. The provided agent in the Basic_Agent class below always selects a random action for each of the four rotors. These four speeds are returned by the act method as a list of four floating-point numbers.

For this project, the agent that you will implement in agents/agent.py will have a far more intelligent method for selecting actions!

import random

class Basic_Agent():
    def __init__(self, task):
        self.task = task
    
    def act(self):
        new_thrust = random.gauss(450., 25.)
        return [new_thrust + random.gauss(0., 1.) for x in range(4)]

The Task¶

A sample task has been provided for you in task.py. Open this file in a new window now.

The __init__() method is used to initialize several variables that are needed to specify the task.

  • The simulator is initialized as an instance of the PhysicsSim class (from physics_sim.py).
  • Inspired by the methodology in the original DDPG paper, we make use of action repeats. For each timestep of the agent, we step the simulation action_repeats timesteps. If you are not familiar with action repeats, please read the Results section in the DDPG paper.
  • We set the number of elements in the state vector. For the sample task, we only work with the 6-dimensional pose information. To set the size of the state (state_size), we must take action repeats into account.
  • The environment will always have a 4-dimensional action space, with one entry for each rotor (action_size=4). You can set the minimum (action_low) and maximum (action_high) values of each entry here.
  • The sample task in this provided file is for the agent to reach a target position. We specify that target position as a variable.

The reset() method resets the simulator. The agent should call this method every time the episode ends. You can see an example of this in the code cell below.

The step() method is perhaps the most important. It accepts the agent's choice of action rotor_speeds, which is used to prepare the next state to pass on to the agent. Then, the reward is computed from get_reward(). The episode is considered done if the time limit has been exceeded, or the quadcopter has travelled outside of the bounds of the simulation.

In the next section, you will learn how to test the performance of an agent on this task.

The Agent¶

The sample agent given in agents/policy_search.py uses a very simplistic linear policy to directly compute the action vector as a dot product of the state vector and a matrix of weights. Then, it randomly perturbs the parameters by adding some Gaussian noise, to produce a different policy. Based on the average reward obtained in each episode (score), it keeps track of the best set of parameters found so far, how the score is changing, and accordingly tweaks a scaling factor to widen or tighten the noise.

Amend task.py to specify a task of your choosing. If you're unsure what kind of task to specify, you may like to teach your quadcopter to takeoff, hover in place, land softly, or reach a target pose.

After specifying your task, use the sample agent in agents/policy_search.py as a template to define your own agent in agents/agent.py. You can borrow whatever you need from the sample agent, including ideas on how you might modularize your code (using helper methods like act(), learn(), reset_episode(), etc.).

Note that it is highly unlikely that the first agent and task that you specify will learn well. You will likely have to tweak various hyperparameters and the reward function for your task until you arrive at reasonably good behavior.

As you develop your agent, it's important to keep an eye on how it's performing. Use the code above as inspiration to build in a mechanism to log/save the total rewards obtained in each episode to file. If the episode rewards are gradually increasing, this is an indication that your agent is learning.

"""
Function used to train an agent,
it returns:
- The total reward per episode during the training.
- The information (quadcopter status) of the last run.
- The information (quadcopter status) of the best run.
- The information (quadcopter status) of the worst run.
"""
def train_agent(task, agent,
                max_episodes=500,
                episode_reward_threshold=None, ten_avg_reward_threshold=None,
                action_to_rotor_speeds=None):

    reward_results = {x : [] for x in reward_labels}

    worst_episode_reward = float("inf")
    best_episode_reward = float("-inf")

    last_run_data = None
    best_run_data = None
    worst_run_data = None
    last_10_episode_rewards = []

    for i_episode in range(1, max_episodes+1):

        last_run_data = []

        state = agent.reset_episode() # starting a new episode
        episode_reward = 0
        while True:
            action = agent.act(state) # getting the action to perform from the agent

            # converting an action to rotor speeds, if a convertion function was provided
            if action_to_rotor_speeds is not None:
                rotor_speeds = action_to_rotor_speeds(action)
            else:
                rotor_speeds = action

            # updating the rotors speeds (this updates the simulator as well)
            next_state, reward, done = task.step(rotor_speeds)

            episode_reward += reward # updating episode's total reward

            
            last_run_data.append([task.sim.time] + list(task.sim.pose) + list(task.sim.v) + \
                                 list(task.sim.angular_v) + list(rotor_speeds))

            agent.step(action, reward, next_state, done) # agent learns

            state = next_state # updating next state

            if done:
                break

        # reporting the total reward of the episode
        reward_results['episode'].append(i_episode)
        reward_results['reward'].append(episode_reward)

        # storing the total reward of the last 10 episodes, in order to display the average
        if (len(last_10_episode_rewards) >= 10):
            last_10_episode_rewards.pop(0)
        last_10_episode_rewards.append(episode_reward)
        avg_last_10_episode_rewards = np.mean(last_10_episode_rewards) \
                                      if len(last_10_episode_rewards) == 10 else float('NaN')

        # updating best and worst episode rewards
        if episode_reward > best_episode_reward:
            best_episode_reward = episode_reward
            best_run_data = last_run_data # storing the best run
        if episode_reward < worst_episode_reward:
            worst_episode_reward = episode_reward
            worst_run_data = last_run_data # storing the worst run

        # information to display/debug during the training
        print("\rEpisode = {:4d}, reward = {:8.3f} (best = {:8.3f}, worst = {:8.3f}, avg_last_10 = {:8.3f})"
              .format(i_episode, episode_reward,
                      best_episode_reward, worst_episode_reward, avg_last_10_episode_rewards),
              end="")
        sys.stdout.flush()

        # if episode's reward reaches a satisfactory theshold, terminating the training 
        if episode_reward_threshold is not None and \
           episode_reward >= episode_reward_threshold:
            break
        if ten_avg_reward_threshold is not None and \
           not math.isnan(avg_last_10_episode_rewards) and \
           avg_last_10_episode_rewards >= ten_avg_reward_threshold:
            break

    return reward_results, last_run_data, best_run_data, worst_run_data

def write_run_data(run_data):
    with open(file_output, 'w') as csvfile:
        writer = csv.writer(csvfile)
        writer.writerow(labels)
        for i in range(len(run_data)):
            writer.writerow(run_data[i])


"""
Gathers the results from an episode's run and organize it by label,
aiming to use it to plot the quadcopter status during the episode.
"""
def retrieve_results_from_run_data(run_data):
    results = {x : [] for x in labels}
    for i in range(len(run_data)):
        for j in range(len(labels)):
            results[labels[j]].append(run_data[i][j])
    return results

Reflections¶

Question 1: Describe the task that you specified in task.py. How did you design the reward function?

Answer:

I chose the task "Hover in Place", which consist in making the quadcopter to stay in the same position, i.e. not moving to another position of the quadcopter spining in any direction.

I actually didn't overwrite the file task.py, instead I created two new files tasks/hover_in_place.py for doing the task for exploration, and tasks/hover_in_place_with_sync_rotors.py for the definitive task. In the definitive task there is a restriction for the rotors to rotate at the same speed, since it was observed during several experimentation steps that the best results (i.e. where the quadcopter did't move and spin too much) where achieved when the speeds of the rotors were very similars.

For exploration, I initialy designed a reward function that penalized the distance from the initial position and spins of the quadcopter (i.e. changes of the angles and angle-speeds), my first try was using a function like:

math.exp(-(
  0.5 * ((agent.sim.pose[:3] - agent.target_pos)**2).sum() +
  0.5 * (agent.sim.pose[2] - agent.target_pos[2])**2  +
  0.001 * (agent.sim.v**2).sum() +
  0.001 * (agent.sim.angular_v**2).sum()
))

The idea of the exponential function was to limit the reward in the range (0,1], trying to get a positive reward if the agent kept flying, however the algorithm was having troubles using the exponential function for optimization because beyond certain distances the reward was negligible, making the algorithm to follow a learning path where the quadcopter was made to fall to the ground.

Then the second attemp was a function of the form:

1.0 - 0.5 * ((agent.sim.pose[:3] - agent.target_pos)**2).sum()
    - 0.5  * (agent.sim.pose[2] - agent.target_pos[2])**2
    - 0.001 * (agent.sim.v**2).sum()
    - 0.001 * (agent.sim.angular_v**2).sum()

This function involves negative rewards, which at some point are worse than zero (i.e. episode terminated because quadcopter felt). Still the agent had problems to learn, but at lest allowing the exploration of different states during training and not following a learning path that made the quadcopter to fall, this allowed to perform observations that allowed to do a simplification of the task.

It is also worth to mention that I tried other reward functions like te following that goes to negative infinite the greater the distance to the desired position is, and to infinite the closer to the desired position it gets:

-math.log(math.sqrt((agent.sim.pose[:3] - agent.target_pos[:3]).sum()**2))

And the following variant using tanh to restrict the rewards in the range (-1, 1):

math.tanh(-math.log(math.sqrt((agent.sim.pose[2] - agent.target_pos[2]).sum()**2)))

The reward function for the definitive task "Hover in Place with rotors restricted to rotate at the same speed" that worked the best is the following. It uses a base value of 1.0 to reward the agent just for keeping the quadcopter flying, then a penaliziation is contemplated for being away of the initial position, only the z position is needed since being at the same speed doesn't change the other coordinates, and an extra penalization for changes in the velocity on the z-coordinate, squares of these magnitudes are used since in this way the penalization gets smaller if the the quadcopter gets closer to the initial position and getts bigger if the distance gets bigger. Note that penalizations for spins (i.e. changes in the angles or angle-velocities) are not necessary since these are not a problem when all rotors have the same rotational speed:

1.0 - 0.5  * (agent.sim.pose[2] - agent.target_pos[2])**2 - 0.5 * (agent.sim.v[2]**2).sum()

I used Deep Deterministic Policy Gradients (DDPG) for the task, since this kind of method suits for continous states/actions, and a lot of directions are provided in the course's material. I tried different neural network architectures and tweking different hyper-parameters till the agent was able to show some indications of learning. The best results were obtained when I did a simplification of the task (i.e. restricting the rotors to be at the same rotational speed), then the agent was able to learn in few episodes with the hyper-parameters described bellow.

The implementation of the agent is in the directory agents/ddpg and contains the following components:

  • Agent (file agent.py)
  • Actor (file actor.py)
  • Critic (file critic.py)
  • ReplayBuffer (file replay_buffer.py)
  • OUNoise (file ou_noise.py)

The final choice of the hyper-parameters is:

  • Gamma: 0.99
  • Tau: 0.02,
  • Capacity of experience replay memory: 100000
  • Size of minibatch from experience replay memory: 64
  • Exploration Mu: 0
  • Exploration Theta: 0.0001
  • Exploration Sigma: 0.0001
  • Actor's Learning Rate: 0.000001
  • Critic's Learning Rate: 0.000001

The agent uses neural network architectures.

  • Actor :
    • Dense(units=256) + L2 Regularisation + ReLu Activation + BatchNorm
    • Dense(units=512) + L2 Regularisation + ReLu Activation + BatchNorm
    • Dense(units=256) + L2 Regularisation + ReLu Activation + BatchNorm
  • Crtic :
    • State pathway:
      • Dense(units=256) + L2 Regularisation + ReLu Activation + BatchNorm
      • Dense(units=512) + L2 Regularisation + ReLu Activation
    • Action pathway:
      • Dense(units=256) + L2 Regularisation + ReLu Activation + BatchNorm
      • Dense(units=512) + L2 Regularisation + ReLu Activation
    • Combining: Add + ReLu Activation

Question 3: Using the episode rewards plot, discuss how the agent learned over time.

  • Was it an easy task to learn or hard?
  • Was there a gradual learning curve, or an aha moment?
  • How good was the final performance of the agent? (e.g. mean rewards over the last 10 episodes)

Answer:

The task seemed to be simple enough at the begining, however during experimentation it proved to be very hard and the agent was not showing signs of being learning, I thing this was because the complexity of the system is high, even small changes in the rotor speeds cause very erratic behaviors in the quadcoper, causing it to spin or drop in altitude (aparently the algorithm was in an optimization path to fall). Also in some experimentation paths, I noticed the algorithm followed a path were some rotors were not even rotating.

There actually was an aha moment, when during the experimentation I analized one of the runs with the best results, then I noticed that the rotors were rotating at similar speeds. Then I decided to do a simplification of the task that restricted the four rotors to be rotating at exactly the same speed, i.e. that reduced the dimension of the actions from 4 to 1. With this simplification the agent was able to learn quickly and with pretty acceptable quadcoper's behavior.

The final performance after 500 episodes, as shown in the information for the last episode's run during the training, was an episode reward of 95.894, the average of the 10 last episodes was 97.091 meaning that the agent was learning consistently as it also can be seen in the reward/episode plot. Looking at the behavior's graph of the quadcoper in the last episode, it seems that it barely moves from the initial position.
